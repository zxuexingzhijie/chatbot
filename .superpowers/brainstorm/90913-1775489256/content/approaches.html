三种实现方案对比

选择最适合"奇幻酒馆探索"MVP的架构方案

A

状态机驱动 + LLM润色

核心思路：用Python有限状态机管理场景/对话流转，规则引擎硬编码世界逻辑，LLM只负责将干巴巴的状态变化渲染成生动文本

Pros

  • 剧情100%可控，不会跑偏
  • LLM调用最少，成本最低
  • 状态可追踪、可调试
  • 离线模式可用（降级为预设文本）

Cons

  • 自由度受限于预设状态
  • 内容创作工作量大
  • 缺乏"惊喜感"
B

混合管道架构（推荐）

核心思路：三层管道 — 意图解析层（LLM分类）→ 规则引擎验证 → LLM叙事生成。规则引擎做"守门人"，LLM做"说书人"

Pros

  • 自由输入+可控剧情的最佳平衡
  • 完全符合两份文档的设计理念
  • 模块化清晰，各层可独立测试
  • 支持渐进增强（先硬编码后AI）

Cons

  • 需要2次LLM调用/轮（解析+生成）
  • 意图分类准确率需要调优
  • 开发复杂度中等
C

全LLM Agent驱动

核心思路：用一个强LLM Agent作为GM（游戏主持人），通过Function Calling操纵世界状态，System Prompt嵌入世界规则

Pros

  • 最大自由度和创意空间
  • 代码量最少，快速出原型
  • Function Calling管理状态可追踪

Cons

  • 剧情容易失控（AI Dungeon老问题）
  • API成本最高（长context）
  • 调试困难，行为不可预测
  • 离线无法运行
