## 交互界面与操作按钮分析（以 AI Dungeon 为例）  
AI Dungeon 的 Web 界面底部有四个主要命令按钮：**“Take a turn”**、**“Continue”**、**“Retry”** 和 **“Erase”**【46†L158-L165】。其中，“Take a turn”按钮让玩家可以在故事中输入并执行一个动作（即玩家回合）【46†L158-L165】；“Continue”则让故事自动推进下去，不需要玩家额外输入；“Retry”用于重新生成最近产生的故事片段；“Erase”用于删除最近的故事输出并返回到之前的状态【46†L158-L165】。这些按钮语义告诉我们，游戏支持玩家主导操作（Take a turn）与剧情自动推进（Continue）两种模式，同时允许回退和重试功能保障体验。  

在命令行环境下，类似功能可以通过指令和交互逻辑实现：例如，可以让玩家输入特定命令（如“skip”或“wait”）触发“继续”行为，或者支持“undo/重做”命令来实现回退。重点是明确“玩家回合”和“系统回合”界面的切换逻辑：玩家输入时进入“Take a turn”模式，系统应暂停自动生成，等玩家输入完毕再进行剧情生成；如果玩家希望剧情自行推进，则跳过玩家输入阶段直接调用剧情生成逻辑。总之，根据AI Dungeon的设计，我们应区分**玩家主动输入阶段**和**系统自动推进阶段**，并提供相应的交互按钮或命令来控制流转（如“玩家行动”与“等待下一步”）。  

## 混合架构与分工设计  
现有AI叙事游戏往往采用**混合架构**：通过预设规则或结构化状态来维持故事一致性，同时调用大语言模型生成文本。研究表明纯粹由LLM驱动的互动故事容易出现“无限即兴”而缺乏整体结构【16†L241-L244】【33†L109-L117】。例如，Intra项目的设计者指出，在LLM辅助的冒险中，需要有“真实世界状态”支撑（比如明确的钥匙是否在玩家手中、门是否已解锁等物理事实），否则生成的故事就像纯叙述而非可解的游戏【16†L241-L244】。另一位开发者Kaub强调，LLM应扮演“游戏主持人”的角色，将玩家输入转化为对预先设计好剧情的行动执行【33†L109-L117】。  

据此，我们建议采用**规则引擎+LLM引擎**的分工模式：  
- **规则引擎**（或剧情引擎）负责：维护世界状态、执行业务逻辑、检查玩家行为是否合法、推动主线情节节点等。它应实现游戏的硬性规则和剧情框架，例如：谁持有什么物品、哪些任务已完成、哪些事件必须发生等。规则引擎还决定在当前场景下玩家可选的动作类别。  
- **LLM引擎**负责：根据玩家输入和当前上下文生成富有表现力的叙事文本、对话或环境描述。LLM根据规则引擎提供的简要状态和指令生成输出，但不直接修改核心状态（除了由规则引擎明确允许的部分）。例如，当玩家说“砍开木门”时，规则引擎判断玩家有斧头等条件是否满足，如果满足则允许此次行为；然后调用LLM描述劈开的场景与可能出现的新细节。  

采用上述混合模式有助于**保证主线不偏离并维护角色一致性**：规则引擎预设的主要剧情节点和条件确保故事按轨迹推进【33†L109-L117】；LLM只负责在每次交互中填充细节和对话，从而兼顾自由度和连贯性。这类似于EmergentMind提出的“生成-过滤”框架：在关键节点用固定模板或规则包裹输出，其他情况则由LLM自由发挥【24†L91-L99】。此外，还可在开发过程中引入简单的AI检查：比如对LLM输出进行关键词匹配或事实校验，确保其未违背游戏规则。  

## 命令行输入解析与意图识别  
在CLI环境下，玩家输入自然语言命令后，需要将其解析为游戏可以识别的动作。常见做法包括：  
- **关键词/模板匹配**：预定义常见动词及其同义词（如“move/go”、“take/grab”、“talk/speak”等），使用正则或模式匹配提取意图和对象。例如，“go north”或“walk north”都视为移动命令，进行路径更新。  
- **意图分类模型**：可利用NLP分类器将输入归入预设的意图类别（如移动、交谈、使用物品、查看、攻击等）。例如，让模型选择一个“技能”（Skill）和参数【24†L128-L132】。NetPlay等系统就将玩家输入分解为离散技能调用，LLM仅负责选择技能。这种方式有助于将无限文本收敛到有限交互集，便于规则引擎处理。  
- **LLM辅助解析**：可以在玩家输入后调用LLM短提示（Prompt）让其提取意图，如“用户输入描述了哪种行为？请回答‘move/speak/take/etc.’”。然后根据LLM判定的意图执行相应代码。这种方法需要设计小型提示（few-shot）或使用少量标记数据训练简单分类器。  

为实现**开放输入与可控性兼顾**，建议采用**“意图+自由描述”**的双轨策略：系统识别出玩家的动作类别后，再将完整输入提供给LLM以生成细节描述。如玩家输入“对守卫说谄媚的话”，则解析为“talk”动作，并将原文或相似提示给LLM，让其撰写守卫的反应和对话文本。对用户输入不合理/越界时（例如编造完全不符世界设定的新名词），系统应拒绝并提示（“这样的行为目前无法执行”），或要求重述（“我不明白你的意思”）。同时，可以设计一个可选“提示”命令，让玩家主动获取建议，而非自动给出选项。这样玩家保持自由输入的感觉，又有辅助工具减轻挫败感。  

## 存档与世界状态快照设计  
要实现**持续世界状态**，需要设计结构化的存档格式和快照机制。通常的做法是将游戏的所有关键信息序列化到文件或数据库中。建议存储字段包括：  
- **角色状态**（角色属性、当前地点、拥有物品列表等）。  
- **世界/场景状态**（各地图位置的已发生事件、门是否开启、陷阱状态等）。  
- **任务进度**（主线任务和支线任务的完成阶段、触发的剧情节点）。  
- **事件日志/时间线**（已发生的重要事件序列，用于回顾或记忆系统）。  

例如，可以使用JSON表示当前状态：  
```json
{
  "player": {"location": "TownSquare", "inventory": ["lantern", "key"], "hp": 10},
  "world": {"DoorToDungeon": "locked", "MonsterAlive": true},
  "quests": {"MainEscape": "started", "FindKey": "completed"},
  "timeline": ["found map", "opened gate", "met old man"]
}
```  
这其中，“timeline”记录事件顺序，可帮助LLM生成时引用过去发生的事情【4†L120-L127】【18†L40-L47】。Emily Short也指出，要让游戏中的记忆系统起作用，需要**系统化地存储所有重要事实**【18†L40-L47】【18†L45-L49】。因此，凡是影响后续决策的事件都应在存档中有标记（如完成某个谜题、与NPC达成盟约等）。存档时可采用增量自动保存（如每完成一段情节自动写入文件），加载时则完全恢复所有字段。  

对于主线支线分支多、进度复杂的系统，可以考虑用数据库表或键值存储来管理状态，如一张“Quests”表记录每个任务的阶段，一张“Items”表记录物品归属。一种简化做法是每次剧情节点推进时保存**状态快照**：记录当前完整状态，在返回或重试时直接恢复，以支持“撤销”操作和历史回溯。  

## MVP 实现方案与开发清单  
结合以上设计，一个资源受限的最简可行产品（MVP）可以按以下优先级开发：  

1. **基础框架和CLI界面**：构建Python或其他语言的命令行程序，实现用户输入（文本）和输出（打印文本）。先设定最小可玩世界（如一个房间谜题），并硬编码部分主线进程。  
2. **动作解析与规则引擎**：实现基本命令解析（移动、检查、对话等）。定义主线任务的硬性条件（如需要钥匙才能开门），在游戏引擎中实现这些检查逻辑。先不使用AI时，可用预设文本响应让流程通畅。  
3. **LLM 文本生成接口**：接入一个语言模型API（如OpenAI GPT-4），用于生成环境描述和NPC对话。在玩家输入后，构建Prompt包含当前状态摘要和玩家动作，让LLM输出富文本描述。需要设计Prompt模板以控制生成风格、保持世界一致。  
4. **存档与恢复功能**：实现状态的序列化保存功能（JSON或简单数据库），支持玩家随时存档和加载。确保游戏引擎在恢复后可以继续未完剧情。  
5. **用户输入流程控制**：根据“Take a turn / Continue”逻辑，区分玩家行动阶段和自动推进阶段。添加相应命令如`continue`使剧情自动演进，或`undo`命令允许回退到上一步。  
6. **内容和支线扩充**：MVP后期可根据需要增加支线任务、NPC对话、场景细节等，用AI辅助生成或填充。此阶段还可优化UI（如增加颜色、简易提示）。  

按照这个清单开发，核心目标是让一个简易故事框架跑通：玩家可以输入命令探索世界、交互，并且状态持续保存。初版只需几处交互和LLM接口，加快迭代。一旦框架稳定，再逐步加入更多内容和机制，如人物关系追踪、复杂事件等。务求先让系统可玩且持续，再追求体验丰富。
